# 第 6–10 关（第二世界 2-1 ~ 2-5）关卡设计与开发计划

第一阶段（1-1~1-5，见 [LEVELS.md](LEVELS.md)）已完成并通过多轮实测修复。本篇是第二阶段的**玩法设计 + 工程计划**，对应 [PLAN.md](PLAN.md) 新增的 **Phase 12：第二世界内容制作**。

> 设计原则延续第一阶段：每关只重点教**一个**新机制；双人协作是加分项而非门槛（单人必须能通主线）；每关末尾保留"管道小节 + 水下奖励区"传统并继续做出差异；关卡数据尽量只靠 JSON 扩展表达，新机制以"一个小实体类 + 一段 GameScene 装配代码"为单位，避免大改引擎。

## 0. 设计基准（以当前引擎实测值为准）

沿用第一阶段结论，坐标一律以"格"（tile）为单位（当前实现 1 格 = 96px，即 32px 设计值 × WORLD_SCALE 3）：

| 参数 | 当前实测值 | 设计约束 |
|---|---|---|
| 标准跳跃 | 最高 ~3.5 格，水平 ~5 格（助跑 ~6 格） | 需要跳上的台阶高差 ≤3 格；缺口 ≤5 格 |
| 砖块/悬空条净空 | 底部距可站立面 ≥2 格（192px） | 大形态 1.75 格高，2 格是通行下限（已有全关审计脚本） |
| 有效缺口宽度 | ≥2 格 | 1 格缺口角色会"骑"在两侧掉不下去 |
| 管道间距 | 陷阱管道中心距 ≥8 格 | 第一阶段用户反馈定下的标准 |
| 水下物理 | 跳高 = 陆地 2×，速度 = 陆地 75% | 水下缺口/平台间距按此重算，不能照抄陆地数值 |
| 重生保护 | 掉命重生/泡泡救援后 2s 无敌 | 检查点附近仍避免刷怪堆叠，无敌只是兜底 |
| 形态/分数/金币 | 跨关携带，掉命才重置形态 | 后期关默认按"玩家大概率持有火形态"校准难度 |
| 移动平台 | 单向（仅顶面实心）、乘坐为吸附式 | 新增的传送带/碎裂平台沿用同一套"顶面吸附"思路 |

**硬性检查**（每关建完必须跑）：`可达性 BFS + 净空审计`（第一阶段的临时脚本要在 Phase 12.0 落库成 `tools/audit-levels.mjs`）+ Playwright 真实走位回归。

## 1. 第二世界总览

第一世界的主题是"晴朗户外→城堡"，机制轴是**踩踏/踢壳/平台/机关门/管道**。第二世界整体转向"环境本身就是敌人"：每关的新机制都是一种**改变移动手感的环境力**（冰、暗、传送带、风），最后在双 Boss 战里全部收拢。

| 关卡 | 主题 | 尺寸（格） | 本关新机制 | 新敌人 | 敌人配置 | Checkpoint | 双人协作点 |
|---|---|---|---|---|---|---|---|
| 2-1 雪晶滑原 | 雪原冰湖 | 200×12 | **冰面**（低摩擦滑行） | 冰晶龟壳（壳速更快） | Mochi×6 + Shell×3 + 冰壳×3 | x=68 / x=138 | 冰面双踩板宝箱 |
| 2-2 萤光洞窟 | 黑暗洞穴 | 200×14 | **黑暗+光圈视野** | 洞窟蝙蝠（俯冲） | Mochi×4 + Shell×2 + 蝙蝠×6 | x=66 / x=140 | 双光圈照明互补 |
| 2-3 熔岩工厂 | 机械/岩浆 | 210×12 | **传送带 + 喷火柱 + 岩浆池** | 发条齿轮怪（快速巡逻） | 齿轮×6 + Shell×3 + 蝙蝠×2 | x=70 / x=150 | 停机开关掩护过带 |

## 2-1｜雪晶滑原（Frost Slide Fields）

**定位**：第二世界开场关。唯一新概念是**冰面**——踩上去加速度骤降（约为正常 25%），松手不会立刻停下而是滑行，起跳后惯性保留。先在无危险的平地教滑行，再和缺口、敌人、管道组合。

**新机制规格——冰面（iceZones）**：
- JSON：`"iceZones": [{"fromTile": a, "toTile": b}]`，与 waterZones 同构；范围内的**地面顶面**渲染一层淡蓝高光条
- 实现：GameScene 每帧检查玩家脚下 x 是否在冰区且 onFloor → `player._onIce = true`；Player.update 中加速度乘 `ICE_ACCEL_SCALE = 0.25`。现有"目标速度 + 加速度限幅"的写法天然产生滑行/漂移，**不需要额外的摩擦模拟**
- 敌人/蘑菇不受冰面影响（保持可预测），只有玩家滑——这是难度来源也是趣味来源

**分段坐标表**：

| 区间（格） | 内容 | 设计意图 |
|---|---|---|
| 0–14 | 普通平地，一排砖块+金币砖热身 | 与 1-1 开局同构，给"回家感" |
| 14–34 | **第一块冰面**（14–34），平坦无敌人，金币摆成刹车距离标尺（每 4 格一枚） | 安全教学：感受滑行距离，金币本身就是教具 |
| 34–48 | 普通地面，Mochi×2（x=38、44） | 对照组：离开冰面手感立刻恢复 |
| 48–66 | 冰面 + 2 格缺口×2（x=52、60），缺口前金币画出"提前起跳"弧线 | 组合考验 1：冰面上的跳跃时机要提前 |
| 66–70 | 检查点（x=68）+ 问号块（蘑菇，x=67,y=6） | 半程补给 |
| 70–96 | 冰面长廊，冰晶龟壳×2（x=76、88）——踢壳后壳在冰上**不会减速**，可清场也可害死自己 | 组合考验 2：教"冰面踢壳"的双刃剑 |
| 96–118 | 高低平台阵（平台 tileY 6~8，非冰面），蝙蝠预告位：无敌人，纯跳跃 | 中场休息 + 为 2-2 的纵向节奏铺垫 |
| 118–138 | 冰面 + Mochi×3 交错巡逻（x=122、128、134） | 综合：滑行中控制踩踏落点 |
| 138–142 | 检查点（x=138）| Boss 无，但管道小节前设第二检查点 |
| 142–166 | **双踩板宝箱**（协作支线）：踩板 A x=146、B x=152（都在冰面上，滑行会滑出踩板判定区，双人要"站稳"配合），宝箱 x=149,y=6，奖励星星 | 双人点：冰面站桩比平时更考验沟通 |
| 166–188 | 管道小节：陷阱管×3（x=168、176、184，相位错开），间隙金币×3×2 + Mochi×2 | 延续末尾管道传统，标准 8 格间距 |
| 188–192 | 入口管道（x=190）↔ 水下冰洞（下述） | — |
| 192–196 | 旗杆（x=194） | — |
| 205–235（隔离区） | **水下冰洞**：水域 + 冰面并存——水下跳高 2× 且落地继续滑，金币 8 枚摆成波浪线，出口管道 x=228 | 水+冰叠加是本关送给熟练玩家的"手感玩具" |

**敌人清单**：Mochi×6、ShellBuddy×3（普通）、冰晶龟壳×3
**新敌人——冰晶龟壳（iceshell）**：ShellBuddy 换色（冰蓝+白条纹），唯一差异：壳被踢出后速度 300（普通 220）且撞墙反弹不衰减。工程上是 ShellBuddy 子类改一个常数。
**道具/机关**：金币 ~46 枚；问号块×3（蘑菇/火焰花/星星）；砖块 ~12；双踩板宝箱×1
**美术需求**：雪地地面色替换（白+淡蓝）、冰面高光条、冰壳怪换色、雪花背景粒子（可选）

**验收清单**：
- [ ] 冰面滑行距离：全速松手滑 ~3 格（调 ICE_ACCEL_SCALE 至此手感）
- [ ] 冰面上的 2 格缺口用"提前 1 格起跳"能稳定通过（Playwright 随机相位跳 8 次 ≥6 成）
- [ ] 冰壳被踢出后能连穿 2 只 Mochi 且计分给踢壳者
- [ ] 双踩板在滑行经过时**不触发**（必须站定），单人无法同时踩两板
- [ ] 可达性/净空审计 0 问题

## 2-2｜萤光洞窟（Glowmoss Cavern）

**定位**：视觉规则的关卡——**你只能看见身边一圈**。地形本身难度中等，紧张感全部来自视野管理。双人的天然优势（两个光圈）第一次成为核心体验。

**新机制规格——黑暗与光圈（darkness）**：
- JSON：关卡根字段 `"darkness": {"alpha": 0.92, "lightRadiusTiles": 4.5}`
- 实现：全屏 RenderTexture 铺黑（`setScrollFactor(0)`、depth 高于场景低于 HUD），每帧 clear → fill → 以每个存活玩家为圆心 `erase` 一张径向渐变光斑贴图（世界坐标→屏幕坐标换算）；萤火提灯道具同样 erase。Phaser RT erase 是成熟做法，两三个光源无性能压力
- **萤火提灯（lantern，新道具）**：拾取后 12 秒内该玩家光圈半径 ×1.8，HUD 侧显示倒计时小图标；关内固定点位刷新（不随机）
- 星星无敌期间光圈同样放大（星星本来就该"发光"）——免费的一致性彩蛋

**新敌人——洞窟蝙蝠（bat）**：
- 平时倒挂在指定悬点（`y` 即悬点），玩家水平距离 <6 格且在其下方时俯冲：沿正弦弧线下扑到玩家高度再回升归位，往返 ~2.4s，冷却 2s
- 可踩（俯冲中判定踩踏更容易——鼓励主动迎击）、可火球、侧碰伤人；黑暗中它的眼睛保持两个红点微光（**在黑幕之上渲染**，这是公平性关键：看不见地形可以，看不见威胁不行）
- 工程：Enemy 子类，状态机 hang/swoop/return，无重力（swoop 用 tween 或手写正弦位移 + body.reset 同步）

**分段坐标表**：

| 区间（格） | 内容 | 设计意图 |
|---|---|---|
| 0–12 | 洞口：黑幕从 x=8 开始渐入（alpha 0→0.92 过渡 4 格） | 不搞突然全黑，给适应过程 |
| 12–30 | 平坦通道，金币每 3 格一枚（金币在黑暗中保留 20% 自发光轮廓） | 金币充当"面包屑导航"，这是本关金币的第二职能 |
| 30–34 | 第一盏萤火提灯（x=32,y=7）+ 砖块阵 | 教提灯：拾取瞬间视野豁然开朗的爽点 |
| 34–64 | 蝙蝠×3（悬点 x=40、50、58,y=4），地面 Mochi×2 | 红点微光教学：先听/看红点再决定走位 |
| 64–68 | 检查点（x=66）+ 问号块（蘑菇） | — |
| 68–100 | **纵深平台段**（heightTiles 14 的上半部）：三层平台，上层金币多但蝙蝠密（×2），下层安全但绕远 | 风险/回报分层，黑暗放大了选择的心理成本 |

| 106–136 | 龟壳×2 + Mochi×2 混编，地面有 2 格缺口×2（缺口边缘各摆 1 枚金币标记） | 黑暗中的缺口必须有金币标记，这是公平性铁律 |

| 142–168 | 管道小节：陷阱管×3（x=144、152、160），**夹子在黑暗中同样带红点微光**，间隙金币+Mochi×2 | 管道传统 + 黑暗变体 |
| 168–172 | 入口管道（x=170）↔ 水下萤光湖 | — |
| 172–178 | 旗杆（x=175） | — |

**敌人清单**：蝙蝠×6、Mochi×4、ShellBuddy×2

**美术需求**：黑幕+径向光斑贴图、萤火提灯图标、蝙蝠（挂/飞两帧）、发光金币轮廓、萤光水草

**验收清单**：
- [ ] 黑幕下 60fps/120fps 均无掉帧（RT erase 每帧 ≤3 光源）
- [ ] 全部缺口边缘 1 格内有发光金币标记（脚本断言：遍历缺口查金币距离）
- [ ] 蝙蝠红点在黑幕之上可见；俯冲预兆（红点抖动 0.3s）先于位移
- [ ] 提灯倒计时结束光圈平滑收缩（1s 缓动，不瞬间变黑）
- [ ] 关掉黑幕后用第一阶段全套审计跑一遍（黑暗不改变几何）

## 2-3｜熔岩工厂（Molten Gearworks）

**定位**：节奏关。传送带改变"站着不动=安全"的假设，喷火柱给出严格的时间窗，岩浆池是比深坑更醒目的即死边界。本关是第二世界的"机械难度顶点"（2-4 考操作，2-3 考节奏判断）。

**新机制规格**：
1. **传送带（conveyors）**：`{"fromTile","toTile","tileY","speed","dir"}`（speed 设计值 px/s，dir ±1）
   - 实现：与移动平台乘坐同构——每帧对"站在带面上"的玩家/敌人/蘑菇附加 `dir*speed*WORLD_SCALE*dt` 位移（吸附式，不改 body 速度，避免与输入速度打架）；带面滚动纹理用 tileSprite 的 `tilePositionX` 递增，零成本动画
   - 敌人也受带影响：Mochi 在带上会被动位移——关卡用这一点做"齿轮怪乘带冲脸"的桥段
2. **喷火柱（flameJets）**：`{"x","tileY","periodMs","phaseMs","burnMs"}`
   - 从地面/带面向上喷 3 格高火柱：预热 0.6s（喷口红光+烟粒子）→ 喷发 burnMs（默认 900ms，接触=受击）→ 熄灭。判定用手动 AABB（同 PipeTrap 模式），无敌帧规则一致
3. **岩浆池（lavaZones）**：`{"fromTile","toTile"}`
   - 视觉：橙红波动条 + 气泡粒子；判定：玩家 bottom 触及岩浆面 = 直接 `handlePlayerDown`（等同掉坑，走泡泡/扣命全套规则）。**不是**受击降级——这是本关"深坑"的替代品，地图上所有"坑"都换成岩浆以强化主题

**新敌人——发条齿轮怪（gearmochi）**：Mochi 子类，速度 2×（100 设计值），外观齿轮化（旋转的方齿贴图/程序化方块），可踩可火球。定位：在传送带上顺带冲过来，逼玩家在带上主动跳跃。

## 2-4｜狂风高塔（Gale Spire）——纵向关

**定位**：全游戏第一个**纵版关卡**（引擎的纵向镜头、纵深世界高度均已就绪，1-3 已验证过 16 格高度）。向上攀爬 44 格高的塔，阵风周期性横推，碎裂平台限制"站着等风停"的保守策略。摔落不即死（下层平台会接住），代价是进度回退——挫败感控制在"再爬一次"而不是"扣命"。

**新机制规格**：
1. **阵风（windGusts）**：`{"fromTileY","toTileY","dir","forcePx","periodMs","gustMs","phaseMs"}`
   - 塔身按高度分风带；周期循环：静风 →（前兆 0.8s：飘叶粒子+呼啸音效渐强）→ 阵风 gustMs（对风带内**空中**的玩家施加水平加速度 forcePx；地面上的玩家减半）→ 静风
   - 关键公平性：风只在**前兆之后**生效；HUD 不加风向标，靠粒子方向传达（美术即 UI）
2. **碎裂平台（crumblePlatforms）**：`{"x","y","widthTiles"}`
   - 玩家站上 0.4s 后开始抖动+掉渣，再 0.4s 后碎裂（body 禁用+碎片粒子），3s 后原位重生。敌人站立不触发碎裂（避免蝙蝠把路拆了）

## 4. 实施记录（Phase 12 完成后补充）

**状态：已全部实施并实测通过**（2-1 9项 / 2-2 11项 / 2-3 13项 / 2-4 12项+18跳 / 2-5 12项含十关全链路；第一世界回归 17/17 无波及）。各关最终数值以 `public/assets/maps/2-*.json` 为准（2-4 上半塔在实测中重构过，与本篇初版坐标不同）。

**实施中踩出的新教训**（后续建关必须对照，已在审计/实测中反复咬人）：
1. **起跳走廊上方 ~5.5 行内不得有任何实心平台**：跳跃冲量固定（无变高跳），满跳头部上升 3.5 格 + 身高 1.75 格；折返式阶梯若把上层平台叠在下层跳跃路径的头顶（哪怕 4 行间距），起跳必撞头。折返必须错列，或同列间隔 ≥6 行。审计工具的可达性 BFS 查不出这类"可达但撞头"，要靠实跳测试。
2. **世界顶边界会截断跳跃弧线**：纵版关在 y≤3 行起跳时，头部触到 y=0 即被钳位、滞空时间锐减（4 格间隙从"轻松"变"物理不可能"）。塔顶最后几跳的间隙要按截断后的滞空（~0.33s，约 3 格）设计。
3. **传送带位移不要 `rect.x += shift` 后再调 `body.updateFromGameObject()`**：Body.postUpdate 会把这段位移当成物理运动再叠加回 GameObject，实测恰好 2× 带速。只改 GameObject，让 preUpdate 自行同步。
4. **"组默认值覆盖"再次咬人**：蝙蝠的 `setAllowGravity(false)` 被 `enemyGroup.add()` 重置，悬挂点缓慢下沉。凡无重力实体入组后必须重申（现约定实体自带 `noGravity` 标志，_loadEnemies 统一处理）。
5. **Boss 竞技场必须处理"战斗中死亡复活在墙外"**：封墙类 Boss 战 + 墙外检查点 = 软锁（墙不开、Boss 还在、人进不去）。2-5 的实现是：场上无存活玩家在墙内时自动拆墙清 Boss、战斗回到待触发态，重新走进去便重新开打。设计封闭 Boss 房时一律要有这类逃生阀。
6. **复活点除了不嵌地面，还不得与管道等实心机关水平重叠**（2-4 露台检查点曾嵌进入口管道；审计规则 4.5 已同时覆盖两类 + P2 偏移落点）。
7. **交替风带没有"双静风窗口"是设计特性**：两条反向阵风相位交错时，全静风窗口可能为零——玩法是"借顺风跳"，自动化测试也必须按这个玩法写（等顺风而不是等无风）。

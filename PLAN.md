# 超级玛丽风格网页游戏 - 开发计划

## 1. 项目定位

- **类型**：横版卷轴跳跃游戏（Platformer），玩法参考经典超级玛丽
- **技术栈**：Phaser 3（游戏引擎）+ Vite（构建工具）+ JavaScript
- **目标**：完整初版 —— 3~5 个关卡，多种敌人，道具系统，计分/生命系统，基础音效音乐
- **输入方式**：键盘 + **游戏手柄（Gamepad）**，两者可随时切换使用
- **人数**：支持 **1~2 人本地同屏合作**
- **美术风格**：Q 版萌系（chibi），非写实像素风
- **IP 提示**：「超级玛丽」的角色、音乐、美术是任天堂的注册商标/版权资产。本计划仅做**玩法致敬**的原创游戏（自绘 Q 版美术 + 原创角色命名与配色，如「跳跳兔」代替马里奥），仅用于个人学习/作品集展示，不做商业发布或使用任天堂官方素材，避免版权风险。如果你确实需要用官方素材做私人练习项目，那是另一个话题，我会在实现时用占位美术/开源素材代替。

## 2. 美术风格定位（Q 版 · 萌）

整体美术基调是"可爱优先"，不是写实像素怀旧风，所有设计都围绕"萌"展开：

- **角色比例**：大头小身的 Q 版比例（约 2.5~3 头身），轮廓圆润饱满，避免尖锐棱角
- **配色**：马卡龙/糖果色系（粉彩蓝、粉、奶黄、薄荷绿），高饱和但柔和不刺眼；主色调刻意避开任天堂经典红蓝撞色
- **角色设计**（区分双人，非单纯换色，保证剪影可识别）：
  - Player 1 = 「跳跳兔」：粉色调兔子造型，长耳朵可作为跳跃时的动态装饰
  - Player 2 = 「喵喵猫」：黄色调猫咪造型，尾巴摆动增加动态感
  - 两者都是大眼睛 + 腮红 + 简单表情（眨眼、张嘴笑、受伤时委屈脸）
- **敌人设计**：拒绝"吓人"设定，全部萌系化改造
  - 巡逻类小怪：圆滚滚"团子"造型，头顶蝴蝶结/触角装饰，替代传统 Goomba 的呲牙形象
  - 乌龟壳类小怪：圆润龟壳配色改为糖果条纹，被踢飞时表情夸张卖萌
  - 受伤反馈用"眩晕星星/爱心"表现，不做写实痛苦表情
- **特效与粒子**：吃道具用爱心/星星/彩带粒子，无敌状态用彩虹渐变描边，避免火焰/爆炸等写实特效
- **UI/HUD**：圆角卡片、可爱字体（如手写体/圆体）、糖果色数值条，图标化生命值（爱心图标代替数字血条）
- **音效基调**：跳跃/吃道具用高音萌系拟声音效（叮/啵一类），BGM 走轻快 8bit/音乐盒风格，避免过于激烈的鼓点

> 美术资产落地建议：先用简单几何占位图（圆形+色块）跑通玩法，角色/敌人的正式 Q 版原画和帧动画放到 **Phase 9** 再统一制作，避免过早陷入美术细节而拖慢核心玩法验证。

## 3. 双人合作玩法设计

- **人数**：默认单人开局（跳跳兔），第二玩家可随时"加入"——按下 P2 对应的按键/手柄按钮即可让喵喵猫从当前镜头范围内一个可爱的"泡泡弹出"动画中登场，中途退出则喵喵猫变回泡泡飘走（不强制从头开始两人模式）
- **镜头**：**共享镜头**（参考《新超级玛丽兄弟》），镜头跟随两个玩家的中点；单人模式下等价于镜头只跟随该玩家，无需为单/双人模式写两套镜头逻辑
  - 需要设置"软边界"：玩家离镜头边缘太远会被镜头边界轻推回来（而不是任由其掉出屏幕），拉开太远时给一个方向指示箭头提示队友位置
- **生命机制**：**共享生命池 + 泡泡复活**
  - 两人共用同一组生命值；某玩家受伤/掉出屏幕时不会立即扣光共享生命，而是变成一个漂浮的可爱泡泡（原地缓慢下沉/漂浮几秒）
  - 队友碰到泡泡即可复活该玩家（复活时放可爱特效，如泡泡"啵"一声破裂+撒星星）
  - 若泡泡存续时间内无人营救，才正式扣减共享生命值；共享生命耗尽才触发 Game Over
  - 单人模式下没有"队友营救"的选项，角色受伤直接按原有单人生命规则处理（不生成泡泡）
- **输入分配**（衔接已有的键盘/手柄 `InputManager` 设计）：
  - Player 1：键盘（方向键/WASD + 空格）为默认，若检测到已连接手柄也可直接用手柄 0
  - Player 2：优先分配给第二个已连接手柄；如果只有一个手柄或没有手柄，则退化为键盘副按键位（如 IJKL + U）
  - 手柄/键盘的连接与"加入/退出"事件都要在 UI 上有明确提示（例如画面角落显示"P2 已加入 🐱"）

## 4. 技术栈细节

| 模块 | 选型 | 说明 |
|---|---|---|
| 游戏引擎 | Phaser 3 | 内置 Arcade Physics、Tilemap、Sprite 动画、音频管理 |
| 构建工具 | Vite | 秒级热更新，打包简单，适合静态部署 |
| 关卡编辑 | Tiled Map Editor | 可视化拼关卡，导出 JSON 供 Phaser 加载 |
| 语言 | JavaScript（ES Modules） | 团队/个人熟悉度高，先不引入 TS 增加复杂度 |
| 本地存储 | IndexedDB（分数/历史最佳）+ localStorage（设置/关卡解锁进度） | 详见下方"本地数据持久化方案" |
| 部署 | 静态托管（Vercel / GitHub Pages / Netlify） | Phaser 打包产物是纯静态文件 |

### 本地数据持久化方案

游戏跑在纯静态网页里，没有服务器，所有存档只能落在浏览器本地。评估过 SQLite（需要 sql.js/wa-sqlite 这类 WASM 方案模拟，还要额外接 IndexedDB 或 OPFS 才能持久化，体积和复杂度对这个体量的游戏是过度工程），最终按数据形态分两条腿走：

- **IndexedDB**：存"分数历史"和"历史最佳"这类会持续增长、需要结构化查询的数据，原生浏览器 API，无需引入 WASM 依赖。规划两个 object store：
  - `scoreHistory`：每次通关一条记录 `{ id(自增), levelId, players(1|2), score, coinsCollected, timeMs, playedAt }`，用于"历史成绩列表"
  - `bestByLevel`：以 `levelId` 为主键，记录 `{ levelId, bestScore, bestTimeMs, achievedAt }`，用于"历史最佳"展示和排行榜式对比
  - 可选引入轻量封装库 `idb`（体积约 1KB gzip，只是把原生 IndexedDB 的回调式 API 包装成 Promise，不引入额外复杂度）简化读写代码
- **localStorage**：存体积小、无需查询、同步读取更方便的数据，例如已解锁关卡列表、音量设置、上次选择的语言/难度等
- 统一封装成 `systems/SaveManager.js`，游戏逻辑只调用 `SaveManager.recordLevelResult(...)` / `SaveManager.getBestByLevel(levelId)` 等接口，不直接操作 IndexedDB/localStorage，方便后续如果要换存储方案（比如加云同步）时只改这一层

## 6. 分阶段开发计划

### Phase 0：工程初始化（0.5 天）
- `npm create vite@latest` 搭建项目 + 安装 Phaser
- 配置 `main.js` 基础 Game Config（分辨率、物理引擎、场景列表）
- 跑通一个空场景，确认热更新与部署链路（先部署一个空壳到 Vercel/GitHub Pages）

### Phase 1：单人角色核心手感与输入系统（1.5~2.5 天）
- 加载占位精灵图，实现 `Player` 左右移动、跳跃、重力、落地判定
- 搭建 `InputManager`：统一封装键盘（方向键/WASD + 空格）、**手柄**、预留触屏虚拟按键，对 `Player` 只暴露 `left/right/jump/run/down` 等逻辑状态，玩法代码不关心具体输入设备
- **手柄支持**（基于 Phaser 内置 `Phaser.Input.Gamepad`，浏览器层是标准 Web Gamepad API）：
  - 需要用户先按一次手柄按键"唤醒"浏览器授权（Gamepad API 限制，无法自动检测）
  - 左摇杆/方向键 D-Pad 控制左右移动，摇杆做死区处理（deadzone）避免漂移
  - A/×（Xbox/PS 手柄按钮映射不同，需做一层按钮语义映射，如 `JUMP = pad.A || pad.B`）控制跳跃，肩键做"跑步/加速"
  - 监听 `gamepad.connected` / `disconnected` 事件，断线时无缝回退到键盘，UI 提示当前输入设备
  - 兼容主流手柄布局（Xbox 手柄、PS 手柄、Switch Pro 手柄），优先测试 Xbox 布局作为默认映射基准
- 键盘和手柄同时接入时互不冲突，任意一方输入都立即生效（不需要切换模式）
- 相机跟随主角（`Camera.startFollow`）
- 这一步是整个手感的核心，需要反复调参（跳跃高度、移速、加速度/摩擦力、摇杆死区大小）直到"操作顺手"

### Phase 2：双人合作机制（1.5~2 天）
- `InputManager` 扩展为多玩家分配：P1 默认键盘/手柄 0，P2 优先分配第二个手柄，否则退化为键盘副按键位（IJKL + U）
- 实现"中途加入"：监听 P2 对应设备的首次输入，触发喵喵猫从泡泡中弹出登场；P2 长时间无输入或主动退出则变回泡泡飘走
- `CoopManager`：
  - 共享镜头 —— 镜头目标点 = 两玩家中点，带"软边界"逻辑（超出可视范围的玩家被镜头边界轻推回，同时给出指向队友的方向箭头提示）
  - 共享生命池 —— 统一维护一份生命值，供 `Player` 受伤逻辑调用
  - 泡泡复活 —— 玩家受伤/掉出屏幕时生成 `Bubble` 实体（缓慢下沉+存续倒计时），队友碰撞泡泡即复活；倒计时结束未获救才真正扣减共享生命
- 单人模式复用同一套 `CoopManager` 逻辑（镜头只跟随一人、受伤直接扣命不生成泡泡），避免维护两套分支代码

### Phase 3：关卡与地图系统（1~2 天）
- 用 Tiled 制作第一张测试关卡（地面、平台、缺口、旗杆终点）
- Phaser 加载 Tilemap JSON，设置地面图层的碰撞
- 实现关卡边界、掉出屏幕即死亡判定、到达旗杆触发过关

### Phase 4：敌人与交互（Q 版重塑）（2~3 天）
- `Enemy` 基类：巡逻移动、碰到边缘/墙壁转向
- Mochi（团子怪）：被踩扁消失（压扁成小圆饼+冒星星特效），侧面碰撞玩家受伤
- ShellBuddy（龟壳怪）：被踩后缩壳卖萌，壳可被踢飞撞击其他敌人
- 玩家受伤状态机：无敌闪烁帧、扣血/掉道具、双人模式下改为生成 `Bubble`（见 Phase 2）

### Phase 5：道具系统（1~2 天）
- 金币：计分 + 音效
- 蘑菇：小变大，碰撞体积/精灵切换
- 火花（变大后触发）：可发射弹道攻击敌人
- 星星：限时无敌 + 彩虹描边特效
- 道具从问号砖块中弹出的交互（顶砖块触发）

### Phase 6：计分/生命/HUD/存档（1.5~2 天）
- `HUDScene` 叠加层：分数、金币数、共享生命（爱心图标）、关卡计时；双人模式下需同时显示两名玩家的状态与"P2 已加入"提示
- 共享生命耗尽 -> `GameOverScene`；通关 -> 结算画面，展示本次成绩 + 历史最佳对比，并提供下一关入口
- `ScoreManager` 统一管理分数增减逻辑（双人共享分数，减少"抢分"式竞争，强化合作感）
- `SaveManager` 落地：搭建 IndexedDB 的 `scoreHistory`/`bestByLevel` 两个 object store，通关时调用 `recordLevelResult()` 写入历史记录并更新最佳成绩；结算画面读取 `getBestByLevel()` 做"打破记录"的提示反馈（比如出现小彩带特效）

### Phase 7：音效与音乐（0.5~1 天）
- Phase 1 已提前搭好 `systems/AudioManager.js` 并接入跳跃/落地两个音效，用 Web Audio API 实时合成短促"叮/咚"音调（无需真实音频文件），验证了触发时机没问题；本阶段需要决定是保留合成音效（其实和"萌系 8bit"调性很搭）还是换成正式录制/购买的音频素材
- BGM 循环播放（轻快 8bit/音乐盒风格，切场景时正确停止/切换）
- 吃金币、踩怪、受伤、泡泡复活、通关等其余萌系拟声音效
- 全局静音/音量控制（存 localStorage）

### Phase 8：多关卡内容制作（5 天）
- 前五关的具体主题、机制引入节奏、坐标级分段设计、敌人/道具数值参数、检查点、双人协作点位已细化到 [LEVELS.md](LEVELS.md)，按 1-1 → 1-5 顺序在 Tiled 里建图
- 关卡设计遵循"单关只教一个新机制"的节奏，双人专属机关默认放在隐藏/支线位置，不阻塞单人主线通关（详见 LEVELS.md 总原则）
- `levels.js` 配置驱动关卡加载，避免为每关写重复代码；需支持标记检查点坐标、隐藏协作机关坐标与 Boss 专属字段/阶段参数
- 需实现通用 `CheckpointManager`：记录最近触发的检查点，死亡重生时恢复位置但不重置已收集金币/已触发机关
- 关卡选择界面通过 `SaveManager` 读取 localStorage 中的已解锁/已通关关卡列表，决定哪些关卡可进入
- 每关建完后按 LEVELS.md 各关的验收清单逐条试玩验证，再进入下一关

### Phase 9：Q 版美术资产制作与替换（1.5~2.5 天）—— 已完成（降级方案）
- **实际情况**：当前环境不具备生成真实图片/帧动画素材的能力，因此本阶段改为「用代码把占位几何图形画得更 Q 版萌」，而非替换为正式原画。真正的美术资产替换留到有素材来源时再做。
- 已完成：
  - Player：加了耳朵（兔耳/猫耳）、眼睛+瞳孔（随朝向偏移）、腮红，跟随身体大小/朝向实时更新（`Player.js` `_updateFace()`）
  - Enemy 基类：加了会跟随朝向的眼睛（`Enemy.js` `_updateFace()`），死亡/踩扁时装饰物同步销毁，不留孤立图形
  - Mochi：头顶蝴蝶结；ShellBuddy：糖果条纹壳纹理

- "跳跃碰撞毫无反应"：把 5 关所有砖块+问号砖（共 75 个）逐一用 Playwright 精确模拟"从正下方跳起来碰撞"，碰撞判定和奖励效果（弹出道具/加分/顶碎/小马里奥碰不碎但有反馈）全部正常触发，没有找到真的"完全没反应"的方块；也试过自然的助跑+跳跃去撞一整排连续砖块，行为符合预期（一次跳跃只能碰到跳跃弧线经过的那一块，不会像瞬移测试那样跳一次触发一整排——这是正常判定，不是 bug）。综合两个问题来看，"没反应"很可能是前一个问题的连带效应：砖块显示比角色矮一截，容易误判起跳的时机/位置从而完全跳空／擦边，看起来像"撞了但没反应"，实际是没真正碰到判定框。这次把砖块视觉放大之后，同样的位置应该更容易准确落在判定范围内；如果放出这个版本后还能重现"确实撞到了但没反应"的情况，麻烦告诉我具体是哪一关、哪个位置，我可以针对性再查
- 用 Playwright 验证过：放大后 75 个方块的判定 100% 正常（和放大前一致，物理没被影响）；截图确认放大后视觉比例明显更协调（小/大形态站在砖块旁边都不再显得方块特别矮小）；5 关整体回归无新增问题
- **视觉放大的方向修正（用户反馈：砖块变高后，站在砖块上的玩家脚部看起来陷进砖块里）**：第一版放大是"上下各长一半"（贴图中心不动），结果贴图顶边比物理碰撞体的顶边高出一截，玩家站在砖块上时物理上站在碰撞体顶面、视觉上就"没到砖块顶"，脚踝以下看起来埋在砖块里。改成**只往下长**（贴图顶边和碰撞体顶边严格对齐，中心下移），站在上面脚底完全贴合；多出来的部分垂在砖块下方，砖块本来就悬浮在空中、下面没有别的东西，从下方顶砖时头部与贴图的短暂视觉重叠只有一瞬间（还伴随顶砖的弹跳动画），远不如"一直站着但脚陷进去"显眼。用截图确认站立时脚底和砖块顶边完全平齐

## 7. 关键技术难点预判

1. **跳跃手感调参**：重力、跳跃初速度、土狼时间（coyote time）、跳跃缓冲（jump buffer）需要反复试参数，建议尽早做手感原型
2. **碰撞判定方向**（Phase 4 踩坑实录）：踩怪判定不能用 `body.velocity.y > 0` 判断"正在下落"——Arcade Physics 在碰撞分离阶段会先把 velocity 清零/反弹，等真正调用 collider 回调时这个值已经不可靠，导致所有踩踏曾经被 100% 误判成侧面撞击。正确做法是用 `body.touching.down`（该帧"刚落在什么东西上"的标志）配合"玩家底部 ≤ 敌人顶部"的位置判断。另外，`body.checkCollision.down=false` 这种"关掉某个方向的碰撞"设置是**每个物体自己的**，会连累和地面的正常碰撞一起失效；真正只影响"世界边界"的开关是 `physics.world.setBoundsCollision(...)`（全局的），两者容易搞混
3. **状态机管理**：主角有小/大/带火三种形态 + 无敌/受伤/死亡/泡泡状态，建议用简单状态机（对象+switch 或状态类）而不是一堆布尔标志位
4. **关卡数据与代码解耦**：所有关卡差异应该来自 Tiled 地图数据和配置文件，而不是写死在场景代码里，方便后续加关卡
5. **手柄兼容性**：不同手柄（Xbox/PS/Switch Pro）按钮索引和摇杆轴顺序不完全一致，Web Gamepad API 只在"标准映射"下有一定一致性保证，仍需实机测试常见品牌；手柄需要一次按键交互才能被浏览器检测到，首次进入要有引导提示
6. **双人输入设备动态分配**：手柄热插拔、P2 中途加入/退出时，`InputManager` 与 `CoopManager` 之间的设备归属要保持同步，避免出现"两个玩家抢同一份输入"或"手柄断线后角色失控"的问题
7. **共享镜头的软边界与泡泡复活联动**：镜头拉扯逻辑要考虑到泡泡本身也可能被推出可视范围，需要保证泡泡始终在营救可达范围内漂浮，否则会出现"看不到队友泡泡导致必然团灭"的设计漏洞
8. **IndexedDB 的异步与兼容性坑**：写入是异步操作，通关结算画面要等 `recordLevelResult()` 的 Promise resolve 后再展示"历史最佳"，避免读到旧数据；隐私模式/存储配额受限的浏览器可能禁用或清空 IndexedDB，`SaveManager` 需要做好写入失败的降级处理（比如提示"当前浏览器无法保存记录"，而不是直接报错崩溃）
9. **`physics.add.group()` 的 `defaults` 陷阱**（Phase 8 移动平台踩坑实录）：把一个已经手动 `setAllowGravity(false)`/`setImmovable(true)` 的物体 `group.add()` 进一个用默认参数创建的 Group 时，Phaser 的 `createCallbackHandler` 会用 Group 的 `defaults`（默认 `allowGravity:true`、`immovable:false`）**覆盖回去**，把之前设的值悄悄改掉——现象是"明明设了不受重力却还是往下掉"。正确做法：创建 Group 时就把默认值配好，如 `physics.add.group({ allowGravity: false, immovable: true })`；如果同一个 Group 里的成员需要不同设置（比如金币不受重力但蘑菇要受重力），就得在 `group.add()` 之后手动把该物体自己的设置重新 set 一遍
10. **移动平台不会自动"带上"站在上面的角色**：把平台设成 `immovable` 后用 `velocity` 推动，Arcade Physics 只会做碰撞分离，不会像有些引擎一样自动让站在上面的角色跟着水平/垂直移动。需要每帧手动算平台的位移量（`deltaX`/`deltaY`），判断"角色是否正站在这个平台上"（`body.touching.down` + 位置在平台范围内），再把这段位移显式加到角色身上；直接改 `body.x/y` 会因为不同步时机导致没有效果，改 `rect.x/y` 后调用 `body.updateFromGameObject()` 才是可靠做法
11. **用"瞬间传送"去测试移动中的碰撞体不可靠**：Playwright 自动化测试里为了省事直接 `teleportTo()` 把角色"传送"到平台正上方再等它落下，对静态平台、垂直移动平台都很稳定，但对**水平**移动平台会偶发"直接穿模掉下去"——同样的坐标、同样的参数，这次能落地下次就穿过去，换个全新平台实例测也一样偶发。换成"正常走过去+跳"的自然移动方式后问题消失，说明是 Arcade Physics 对"一帧内被瞬间传送到与水平移动碰撞体正好接触"这种极端情况处理不稳定，不是关卡或平台代码本身的 bug。**结论**：测试移动平台类机制时，优先用模拟按键的自然移动去验证"能不能骑上去"，瞬移只用来测静态定位/触发类逻辑（踩点、检查点、开关等），不要用瞬移测"角色和移动碰撞体的动态交互"
12. **`setScale()` 用在带物理体的对象上会偷偷改碰撞体，导致"越揍越穿地"**：Phase 8 做 Boss（糖果史莱姆王）踩过的最隐蔽的坑——Boss 被踩中但没死时，用 `rect.setScale(1, 0.75)` 做一个"被压扁"的受击反馈，200ms 后再 `setScale(1,1)` 恢复。结果第一下踩踏还正常，紧接着 Boss 就开始无限自由落体穿到地图下方，`body.blocked.down` 永远 false。根源是 Arcade Physics 的 `Body.preUpdate()` **每一帧**都会自动调用 `updateFromGameObject()`，把碰撞体的位置/尺寸按 GameObject 当前的 `scaleX/scaleY` 重新计算一遍——也就是说对一个挂了物理体的 Shape/Sprite 调 `setScale()`，下一帧碰撞体的实际位置和大小就跟着变了，不是纯视觉效果。缩放瞬间把碰撞体的底边"抬"离了地面一点点缝隙，重力立刻把它拽下去，等 200ms 后再恢复缩放时它已经带着下落速度飞出去老远，再也碰不到那层 32px 薄地面。之前排查时先怀疑过"没设 `immovable` 导致被踩踏冲量推穿地面"，加了 `setImmovable(true)` 但完全没解决，才回头定位到真正原因是 `setScale`。**结论**：给挂了 Arcade 物理体的对象做受击反馈，不要用 `setScale()`，用 `setFillStyle()`/换贴图帧/tint 之类不影响碰撞体的视觉手段；如果确实需要缩放视觉效果，要么缩放一个不带物理体的子对象，要么接受"缩放会影响碰撞"并主动用 `body.setSize()`/`updateFromGameObject()` 去补偿
